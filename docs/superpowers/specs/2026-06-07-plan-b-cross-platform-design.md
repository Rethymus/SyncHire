# SyncHire Plan B Cross-Platform Design

## Goal

Ship SyncHire as a lightweight, local-first product across Windows, Linux, and Android without making cloud services a runtime requirement.

## Decision

Plan B uses one static Next.js frontend and two native shells:

- Windows/Linux: Tauri 2
- Android: Capacitor

Both shells consume `frontend/out`, generated by `NEXT_OUTPUT=export npm run build --workspace=frontend`.
User-created application detail and match pages use static query routes (`/applications/detail?id=...` and `/applications/match?id=...`) so local IDs created after packaging do not require dynamic server rendering.

## Architecture

```text
Next.js static frontend
  -> shared local-first product logic
  -> platform adapter layer
     -> browser storage for Lite/Web
     -> Tauri commands or local SQLite for desktop
     -> Capacitor plugins for Android
```

The app must not depend on a remote account, cloud database, telemetry service, or hosted API to complete the core resume/JD/application workflow.

## Platform Strategy

Desktop uses Tauri because it keeps the shell small, uses the system WebView, and defaults toward a narrower native permission surface than Electron. The existing Electron implementation remains useful as a fallback and migration reference, but it is no longer the preferred long-term desktop target.

Android uses Capacitor because the current product is already a web-first interface and Capacitor provides the shortest stable path to APK/AAB packaging. Android should not embed the Python FastAPI service; local persistence should move through Capacitor storage/SQLite/file APIs.

## Security Model

- No required cloud login for local workflows.
- No required cloud sync.
- Native shells load bundled static assets.
- Security headers from Next.js are not available in static export, so desktop/mobile shells own CSP and native permissions.
- External network and filesystem access should be added only through explicit platform adapters.
- Optional AI provider keys must remain local to the device unless the user exports them deliberately.

## Implementation Phases

1. Build static frontend reliably.
2. Add Tauri 2 desktop shell and root scripts.
3. Keep Capacitor Android pointed at the same static output and add build/sync scripts.
4. Replace direct server-only assumptions with local-first platform adapters.
5. Add encrypted storage, import/export, and app-lock controls.
6. Add CI artifacts for Windows, Linux, and Android.

## Acceptance Criteria

- `npm run build:static` produces `frontend/out`.
- `npm run desktop:dev` can launch the app through Tauri during development.
- `npm run desktop:build` has a valid Tauri project to compile.
- `npm run android:sync` can copy the same static assets into Capacitor Android after the Android project exists.
- Core local workflows do not require an internet connection or hosted backend.
